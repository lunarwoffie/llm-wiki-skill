# Large Graph Global/Community Interaction Design

日期：2026-06-18
状态：待用户复核
当前分支：`codex/graph-performance-optimization`
视觉稿：[`docs/spark/mockups/2026-06-18-large-graph-global-community-flow.html`](mockups/2026-06-18-large-graph-global-community-flow.html)

## 目的

这份设计确定 llm-wiki 图谱在 5000+ / 10000+ 节点规模下的产品交互方向。

目标不是把当前图谱做成一个更复杂的自动模式系统，也不是照搬 `graphify` 或 `nashsu/llm_wiki`。目标是明确一条长期路线：

**全局图谱负责看结构，社区聚焦负责读内容。**

当前图谱已经有社区色块、社区图例、筛选、搜索、右侧抽屉、Pin、社区聚焦和密度模式。后续性能优化应围绕这些既有用户动作收敛，而不是增加一套让用户难以理解的自动模式切换。

## 设计结论

采用“全局降级，社区聚焦升级”的双层体验。

1. 全局视角默认支持 5000+ / 10000+ 节点，但只显示地图级信息。
2. 普通节点点击只触发轻量查看，不改变全局视角。
3. 社区色块或社区图例点击是明确的深入动作，进入社区聚焦。
4. 社区聚焦后恢复卡片、标签、关系细节和阅读抽屉。
5. 搜索和筛选不创建新模式，只改变当前全局或社区视角里的可见范围。
6. 缩放只调整细节密度，不自动切换全局/社区主模式。

一句话规则：

**按钮和社区点击决定用户要做什么；缩放只决定当前画多少细节。**

## 为什么不做自动切换

不建议根据缩放自动在“全局大图”和“局部阅读”之间切换。

原因：

- 当前已有社区入口和筛选入口，用户意图已经足够明确。
- 自动切换容易让用户迷失：只是缩放一下，界面突然换成另一套阅读状态。
- 大图性能优化需要稳定的规则，不能让细节组件在缩放边界反复出现和消失。
- 现有社区色块点击已经是自然的深入入口，应继续强化它，而不是绕开它。

允许的自动行为只有信息密度变化：

- 缩远时隐藏普通标签、弱边和复杂效果。
- 放近时显示更多关键标签。
- 手势进行中进一步降级，手势停止后补回允许显示的细节。

## 交互规则

### 1. 全局视角

全局视角是地图，不是阅读界面。

显示：

- 节点点位。
- 社区颜色。
- 少量关键节点标签。
- 重要边或社区间骨架关系。
- 搜索命中和选中节点高亮。
- 社区图例和筛选入口。

隐藏或降级：

- 普通节点卡片。
- 大段文本摘要。
- 全量节点标签。
- 全量边。
- 复杂阴影、动画和长时间运行的动态布局。

全局视角点击普通节点：

- 保持图谱视角不变。
- 高亮该节点和少量直接邻居。
- 右侧打开轻量查看抽屉。
- 不自动进入社区聚焦。
- 不自动打开完整阅读内容。

### 2. 全局轻量查看抽屉

当前代码里点击节点会打开右侧抽屉。这个方向保留，但抽屉需要重新定义为“轻量查看”，不要变成图上大卡片的替代品。

轻量抽屉显示：

- 节点名称。
- 类型。
- 所属社区。
- 连接数。
- 3-5 个最重要相邻节点。
- 1-2 条最强关系。
- “进入所属社区”动作。
- “打开详情 / 阅读”动作。
- 预留“向 Agent 询问这个节点”的扩展位，但不作为第一轮必做入口。

轻量抽屉不显示：

- 长正文。
- 大量相邻节点列表。
- 复杂 Markdown 阅读内容。
- 会迫使全局图谱重排的大组件。

轻量抽屉的定位：

- 它是全局地图的检查面板。
- 它帮助用户决定是否深入。
- 它不承担完整阅读。

### 3. 社区入口

社区色块和社区图例点击继续作为“进入社区”的主入口。

这个设计是合理的，因为用户点击社区本身就是明确动作：我要看这个社区。

进入社区聚焦后：

- 图谱视角 fit 到该社区。
- 只显示该社区内的核心节点和必要关系。
- 其他社区退出主要视野或降为背景。
- 显示卡片、更多标签、阅读抽屉和社区摘要。

普通节点也可以进入社区，但必须通过明确动作：

- 在轻量抽屉里点“进入所属社区”。
- 不建议单击节点直接进入社区。
- 第一轮不把双击作为社区入口；后续只有在实测误触风险很低时再考虑加入。

### 4. 社区聚焦视角

社区聚焦是阅读和理解模式。

显示：

- 社区标题和返回全局入口。
- 社区核心节点卡片。
- 核心关系。
- 社区摘要。
- 关键节点列表。
- 与其他社区的桥接关系。
- 阅读抽屉或详情抽屉。
- “向 Agent 询问这个社区”动作。

规模控制：

- 如果社区本身很大，默认只显示核心节点。
- 其余节点通过筛选、搜索、一跳邻居或“展开更多”进入。
- 不在社区聚焦里一次性恢复所有节点卡片。

### 5. 搜索和筛选

搜索和筛选是对当前视角的收窄，不是模式切换。

全局视角中：

- 搜索命中节点强高亮。
- 命中节点的少量相邻节点轻高亮。
- 非命中节点降淡。
- 右侧轻量抽屉可以显示搜索命中列表。

社区视角中：

- 搜索只在当前社区内优先匹配。
- 可提供“扩展到全局搜索”的动作。

筛选后如果剩余节点少，系统可以逐步恢复更多标签和边，但仍遵守当前视角的主规则。

## 大规模降级规则

全局图谱的目标规模按 5000+ / 10000+ 设计。

这不意味着万级图谱要完整显示全部细节。参考 `graphify` 的实际策略，大规模流畅来自降级和聚合，而不是硬画所有内容。

### 节点

- 5000+：全局默认点位为主，只显示核心标签。
- 10000+：进一步减少标签，只保留搜索命中、选中、核心节点和社区代表节点。
- 节点大小按重要性变化，但普通节点不变成卡片。

### 边

- 默认只显示重要边、强关系边、社区间骨架边。
- 弱边默认隐藏或极淡。
- 拖动、缩放、平移过程中隐藏更多边。
- 停止交互后再补回允许显示的边。

### 标签

- 标签预算按当前可见节点数控制。
- 标签优先级：搜索命中 > 选中节点 > 社区代表节点 > 高连接节点 > 普通节点。
- 普通节点标签只能在悬停、轻量抽屉或社区聚焦中出现。

### 社区

- 社区是全局大图的主要组织单位。
- 当总节点超过阈值时，全局优先展示社区结构。
- 如果完整节点图会影响流畅度，可以提供社区聚合视图。
- 社区聚合视图中，社区本身是节点，社区之间的连接代表跨社区关系强度。

### 动画与物理布局

- 全局图谱不应持续运行高消耗动态布局。
- 初始布局稳定后应停下来。
- 拖动时只更新必要对象。
- 社区聚焦可以允许更丰富的局部动画，但仍要受节点数量限制。

## 技术方向

本设计不在这一阶段锁死具体库，但锁死技术主线：

**一个图谱数据底座，一个全局大图渲染器，一套社区/筛选/搜索规则，细节阅读由抽屉和社区聚焦承接。**

可评估的全局大图渲染方向：

- `Sigma + Graphology`：更偏现代 WebGL 图谱，适合桌面版长期路线。
- `vis-network`：`graphify` 使用的方向，Canvas 图谱成熟，聚合视图策略清晰。
- 当前 DOM/SVG：继续用于小图、社区阅读或离线细节模式，不再要求硬撑万级全局图。

明确不做：

- 不同时引入 `Sigma` 和 `vis-network` 两套全局渲染器。
- 不做三套坐标、三套点击、三套筛选。
- 不把卡片和长文本塞进万级全局画布。
- 不因为引入大图渲染器而丢掉现有阅读抽屉和社区体验。

未来桌面版方向：

- 工作台仍以 React/Vite 为基础。
- 桌面壳优先 Tauri。
- 大图布局计算可放到后台线程或桌面后端。
- 图谱数据、社区检测、搜索和阅读抽屉保持同一产品语义。

## 与参考项目的关系

### graphify

借鉴：

- 大图全局降级。
- 超大图社区聚合。
- 只给关键节点显示标签。
- 交互时隐藏部分边。
- 稳定后停止高消耗布局。

不照搬：

- 不把 llm-wiki 做成静态 HTML 图谱工具。
- 不把完整阅读体验交给图谱画布。
- 不让社区聚合替代社区阅读。

### nashsu/llm_wiki

借鉴：

- 桌面应用方向。
- 成熟图谱库承接大图。
- 布局计算后台化。
- 节点数越大，标签和边越克制。
- 图谱洞察面板的产品思路。

不照搬：

- 不放弃 llm-wiki 现有的社区聚焦、Pin、离线图谱和阅读抽屉。
- 不把图谱只做成结构浏览器。

### 当前 llm-wiki graph-engine

保留：

- 社区色块和社区图例。
- 社区聚焦。
- 筛选和搜索。
- Pin。
- 右侧抽屉。
- 离线 HTML 图谱能力。
- 现有图谱数据语义。

调整：

- 全局视角减少卡片化表达。
- 点击节点的右抽屉改成轻量查看。
- 社区聚焦作为细节阅读主入口。
- 大规模场景引入更强的降级和聚合规则。

## 产品状态机

核心状态只有三类：

1. `global`
   - 全局地图。
   - 支持筛选、搜索、节点轻量查看、社区入口。

2. `global + selected node`
   - 全局地图保持不变。
   - 右侧轻量抽屉显示节点信息。
   - 可以进入所属社区或打开详情。

3. `community focus`
   - 聚焦社区。
   - 恢复卡片、核心关系、社区阅读抽屉。
   - 支持返回全局。

不新增“自动阅读模式”或“缩放触发模式”。

## 抽屉设计

抽屉分两种语义，不必一定是两套组件，但用户感知要清楚。

### 全局轻量抽屉

目的：检查节点，不打断地图。

推荐结构：

1. 标题区：节点名、类型、所属社区、连接数。
2. 关系区：最强关系 1-2 条。
3. 邻居区：最重要相邻节点 3-5 个。
4. 动作区：进入所属社区、打开详情。
5. 扩展区：后续可接入询问 Agent，但不影响第一轮抽屉设计。

视觉要求：

- 窄、轻、信息克制。
- 不使用大段正文。
- 不撑大图谱布局。

### 社区阅读抽屉

目的：帮助理解一个社区。

推荐结构：

1. 社区标题和摘要。
2. 核心节点列表。
3. 内部强关系。
4. 桥接到其他社区的关系。
5. 可向 Agent 发问。
6. 可进一步展开一跳邻居或打开节点详情。

视觉要求：

- 信息可以比全局轻量抽屉更重。
- 允许阅读型内容。
- 仍避免一次塞满所有节点。

## 验收标准

设计层验收：

1. 用户能清楚理解：全局是地图，社区是阅读。
2. 普通节点点击不会让用户突然丢失全局上下文。
3. 社区色块/图例点击作为深入入口合理保留。
4. 右侧抽屉不再被定义为完整阅读入口，而是根据状态承担轻量查看或社区阅读。
5. 5000+ / 10000+ 大图有明确降级规则。
6. 方案没有同时引入多套全局图谱渲染主线。

后续实现验收：

1. 1000 节点全局缩放、平移、悬停、节点点击保持流畅。
2. 5000+ 节点全局视角能稳定浏览。
3. 10000+ 节点至少能通过社区聚合或强降级视图流畅浏览。
4. 全局节点点击只打开轻量抽屉，不自动进入社区。
5. 社区色块/图例点击进入社区聚焦。
6. 社区聚焦后卡片和阅读信息恢复，但受核心节点数量限制。
7. 交互过程中弱边、普通标签和复杂效果可以临时隐藏。
8. 全局图谱不能因为抽屉内容变化而重排大图。

## 后续实现建议

下一步不应直接重写渲染器。

建议先做三件事：

1. 固化性能测量：覆盖 1000、5000、10000 节点的全局浏览和社区聚焦。
2. 定义降级策略：节点、边、标签、社区聚合分别在什么规模下触发。
3. 选型试验：用同一份图谱数据分别验证 `Sigma/Graphology` 与 `vis-network` 的全局大图表现，只保留一个全局渲染主线。

通过试验后再写实现计划。
